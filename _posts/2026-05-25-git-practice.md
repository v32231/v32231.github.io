# 기본 명령어

## git commit

커밋은 Git 저장소에 로컬 디렉토리에 있는 모든 파일에 대한 스냅샷을 기록하는 것이다.

Git 은 커밋할 때마다 디렉토리 전체를 복사하는 것이 아니라 이전 버전과 다음 버전의 변경 내역(delta)을 저장한다.

## git branch

브랜치는 특정 커밋에 대한 참조로, 하나의 커밋과 그 부모 커밋들을 포함하는 작업 내역이다.

브랜치를 많이 만들어도 메모리나 디스크 공간에 부담이 되지 않기 때문에 작업을 작은 단위로 잘게 나눈 브랜치를 사용하는 것이 좋다.

git checkout [브랜치명] 을 사용하면 작업 중인 브랜치를 옮길 수 있다.

## git merge

두 개의 부모를 가리키는 커밋을 만들어낸다. 해당 커밋은 한 부모의 모든 작업내역과 나머지 부모의 모든 작업, 그리고 두 부모의 모든 부모들의 작업내역을 포함한다.

## git rebase

기본적으로 커밋들을 모아서 복사한 뒤, 다른 곳에 저장하는 것이다.

커밋들의 흐름을 보기 쉽게 만들 수 있다는 장점이 있고, 커밋 트리의 히스토리를 수정한다는 단점이 있다.

## HEAD

HEAD는 항상 작업트리의 가장 최근 커밋을 가리킨다. 작업 트리에 변화를 주는 명령어들은 대부분 HEAD를 변경하는 걸로 시작한다. 

일반적으로 HEAD는 브랜치의 이름을 가리키며, 커밋을 했을 때 브랜치의 상태가 바뀌고, 이 변경은 HEAD를 통해 확인할 수 있다.

## 상대 참조

특정 지점에서 출발해서 이동하여 다른 지점에 도달해 작업을 할 수 있다.

**캐럿(^)** : 한번에 한 커밋 위로 이동

**틸드(~) + [num]** : 한번에 여러 커밋 위로 이동

## git reset

브랜치가 예전의 커밋을 가리키도록 이동시키는 방식으로 변경 내용을 되돌린다. 

## git revert

공유 프로젝트처럼 각자의 컴퓨터에서 진행한 로컬 프로젝트의 경우, git reset을 사용할 수 없다. 따라서 그럴 땐 git revert를 대신 사용하면 된다. git revert는 reset과 다르게 커밋 기록은 그대로 두고, 되돌리는 새 커밋을 만든다고 보면 된다.

## git cherry-pick

HEAD 아래에 있는 일련의 커밋들에 대한 복사본을 만들겠다는 명령어이다. `git cherry-pick [commit] [commit] […]` 이런 식으로 여러개의 커밋들의 복사본을 만들 수 있다.

## git interactive rebase

커밋 기록을 직접 편집하겠다는 명령어이다.

이 명령어는 원하는 커밋을 모르는 상황에서 유용하다.

실제 git에서 사용할 경우, 텍스트 편집기에서 파일을 열어 리베이스의 목적지가 되는 곳 아래에 복사될 커밋들을 보여준다. 인터렉티브 리베이스 대화창이 열리면 적용할 커밋들의 순서를 바꿀 수 있고, 원하지 않는 커밋을 빼거나 커밋을 스쿼시할 수 있다.

`git rebase -i` 이런 식으로 -i 옵션을 추가해서 사용해주면 된다.

## git tag

특정 커밋들에 영구적인 이정표(milestone)로 표시한다.

git 태그는 커밋들이 추가적으로 생성되어도 절대 움직이지 않으며, 태그를 체크아웃한 후에는 해당 태그에서 어떠한 작업을 완료할 수 없다.

## git describe

현재 커밋이 가장 가까운 태그로부터 얼마나 떨어져 있는지 설명해주는 명령어이다.

`git describe [ref]` 이런 식으로 사용할 수 있으며 ref에는 커밋을 의미하는 어떤 것이든 쓸 수 있다. ref가 지정되어 있지 않으면 HEAD를 사용한다.

결과는 ‘v0-2-gC2’ 이런 식으로 나오게 되는데 이건 v0이라는 tag로부터 2 떨어진 곳에 c2가 존재한다고 해석할 수 있다.

# 원격 저장소

원격 저장소(git remote)는 또 다른 하나의 컴퓨터에 있는 로컬 저장소의 복사본이다.

원격 저장소는 백업 데이터 저장 역할을 하기도 하고, 다인 프로젝트 진행 시 매우 유용하게 사용된다.

## git clone

원격 저장소의 복사본을 로컬에 생성할 때 사용하는 명령어이다.

git clone 명령어를 사용하면 로컬 저장소에 ‘[remote name]/[branch name]’과 같은 형식의 브랜치가 생기는데 이런 종류의 브랜치를 원격 브랜치라고 한다.

원격 브랜치는 가장 최근에 함께 작업한 원격 저장소의 상태를 반영하며, 체크 아웃 시 분리된 HEAD를 가지게 된다는 특징이 있다. 작업은 항상 다른 곳에서 작업 후 원격 저장소에 작업을 공유하는 식으로 진행해야 한다.

## git fetch

로컬에서 나타내는 원격 저장소의 상태를 실제 원격 저장소의 상태와 동기화한다.

git fetch는 원격 저장소에 있지만 로컬에는 없는 커밋들을 다운 받고, 원격 브랜치가 가리키는 곳을 업데이트를 수행한다.

git fetch는 로컬 파일이나 브랜치를 변경시키지는 않는다.

`git fetch [원격 저장소명(origin)] [장소(브랜치명)]`  이렇게 인자를 넣어서도 사용할 수 있다. 만약 ‘git fetch origin foo’ 라는 명령어를 입력하면 원격 저장소의 foo 브랜치에 있는 현재 로컬에 없는 커밋들을 가져와서 로컬의 ‘origin/foo’ 브랜치 아래에 추가한다.

`git fetch [원격 저장소명(origin)] [원본]:[목적지]` 이렇게 목적지와 원본을 구분해줄 수 있다. git push와 반대라는 걸 기억해 두어야 한다. 목적지에 새 브랜치 이름을 넣어서 새 브랜치를 생성할 수도 있다.

만약 [원본]을 쓰지 않는다면 [목적지]에 적은 이름의 브랜치가 추가된다.

## git pull

원격 저장소에 있는 데이터를 로컬로 내려받은 후에는 다른 브랜치에 있는 일반 커밋처럼 사용할 수 있다.

git pull은 `git fecth; git merge`와 같은 기능을 한다고 볼 수 있다.

‘—rebase’ 옵션을 붙여서 한줄로 커밋 이력을 정리할 수 있다. (하나의 부모를 가리키도록)

참고로 ‘git pull origin foo’ 이 명령어는 ‘git fetch origin foo; git merge o/foo’ 이것과, ‘git pull origin bar:bugFix’ 명령어는 ‘git fetch origin bar:bugFix; git merge bugFix’와 같은 동작을 수행한다.

## git fakeTeamwork

특정 브렌치나 여러개의 커밋이 갱신되는 경우를 표현하기 위한 명령어이다.

`git fakeTeamwork` 또는 `git fakeTeamwork [브랜치명/추가할 커밋 개수]` 이런 식으로 사용할 수 있다.

## git push

로컬에서의 작업을 원격 저장소에 업로드하고, 그 원격 저장소가 새 커밋을 합치고, 갱신하게 한다.

`git push [원격저장소명(origin)] [장소(브랜치명)]` 이런 식으로 선택적으로 인자를 넣어서 사용할 수도 있다. 만약 ‘git push origin main’ 명령어를 입력하면 로컬에 있는 main 브랜치의 모든 커밋을 수집하고, origin의 main 브랜치에 부족한 커밋들을 채워 넣는다.

`git push [원격저장소명(origin)] [원본]:[목적지]` 이렇게 쓰면 원본과 목적지를 다르게 지정할 수 있다. 목적지에 새 브랜치 이름을 넣어서 새 브랜치를 생성할 수도 있다.

만약 [원본]을 쓰지 않는다면 [목적지]에 적은 브랜치가 삭제된다.

이렇게 콜론을 사용한 문법을 콜론 참조스펙(colon refspec) 이라고 한다.

## 원격 저장소 거부

원격 저장소의 main 브랜치가 잠겨있을 경우, 변경사항을 저장하기 위해서는 pull request 과정을 거쳐야 한다.

main 브랜치가 잠겨 있는데 로컬에서 main 브랜치에 커밋한 후 push를 하면 오류 메시지가 뜨게 된다.

## feature branch

feature 브랜치(또는 토픽 브랜치)는 main 브랜치가 아닌 작업을 위해 임시로 만든 브랜치이다.

## 원격 추적

원격 브랜치와 로컬 브랜치는 연결되어 있다.

예를 들어, pull 작업을 하던 도중, 커밋들은 origin/main에 내려받아진 후 main 브랜치로 merge 된다. 이건 push도 마찬가지이다. main 브랜치는 origin/main 브랜치를 추적하도록 설정되어 있기 때문이다.

이런 원격 추적 속성은 git clone 시 자동으로 설정 되는데 이는 사용자가 스스로 임의의 브랜치가 origin/main을 추적하도록 바꿀 수도 있다.

`git checkout -b [임의의 브랜치명] [원격 브랜치명]` 명령어를 사용하면, 새 브랜치를 생성하고, 지정한 원격 브랜치를 추적하게 설정한다. 다른 방법으로는 `git branch -u [원격 브랜치명] [임의의 브랜치명]` 명령어를 사용할 수도 있다. 참고로 임의의 브랜치가 현재 작업하고 있는 브랜치면 임의의 브랜치명은 생략 가능하다.
