# 내용 정리

## 정규표현식 개념

정규표현식은 상태 머신처럼 동작하는 패턴 매칭 도구이다. 한 규칙이 맞으면 다음 규칙과 비교하고, 중간에 실패하면 다시 다른 위치에서 시도하면서 전체 문자열에서 맞는 부분을 찾는 방식이다.

### 문자열 매칭

- 우리가 자주 사용하는 ‘ctrl+F’ 기능과 비슷하다.
- 메타 문자
    
    
    | 표현 | 기능 |
    | --- | --- |
    | \b | 단어 앞 뒤에 다른 글자가 붙어 있지 않은 경우 |
    | ^ | 문자열의 시작을 의미 |
    | $ | 문자열의 끝을 의미 |
    | . | 아무 문자 하나를 의미 |
    | \ | 특수문자의 기능을 없애주는 이스케이프 |
- ‘.’을 잘못 사용하면 여러 취약점으로 이어질 수 있으니 ‘.’을 사용할 때는 주의해야 한다.

### 문자 집합

- 대괄호(’[]’)는 문자 집합을 의미한다.
    - 예시
        
        
        | 표현 | 의미 |
        | --- | --- |
        | [ABC] | 문자 A, B, C |
        | ^[MLB] | M, L, B 중 하나의 문자로 시작을 의미 |
        | [A-L] | A 부터 L 까지의 대문자 중 하나를 의미 |
        | [a-l] | a 부터 l 까지의 소문자 중 하나를 의미 |
        | [0-5] | 0 부터 5 까지의 자연수를 의미 |
        | [A-L0-5] | A부터 L까지의 문자 또는 0부터 5까지의 숫자를 의미 |

### 수량자

- 수량자는 앞에 있는 문자나 패턴이 몇 번 반복되는 지를 지정할 때 사용한다.
- 표현
    
    
    | 표현 | 의미 |
    | --- | --- |
    | + | 하나 이상 |
    | ? | 0 또는 1개 |
    | {n} | n번 반복 |
    | {n,} | 최소 n개 이상 |
    | {n, m} | n개 이상 m개 이하 |
    | * | 0개 이상 |

### 단축 문자

- 단축 문자는 반복되는 문자 집합을 더 짧게 사용하기 위해 사용한다.
- 표현

| 표현 | 의미 |
| --- | --- |
| \d | 숫자를 의미 |
| \s | 공백을 의미. 일반 공백 뿐 아니라 탭 같은 공백 문자도 포함한다. |
| \w | word character로, 알파벳 대소문자, 숫자, 언더스코어를 매칭. |
| \D | 숫자가 아닌 문자를 의미 |
| \S | 공백이 아닌 문자를 의미 |
| \W | word character가 아닌 문자를 의미 |

### 그룹

- 원하는 부분만 추출하기 위해 사용한다.
- 반복되는 패턴을 묶을 때 사용한다.
- ‘()’로 사용 가능하다.

## postMessage 취약점

- postMessage는 로그인 팝업 등 다른 창과 현재 페이지가 메시지를 주고 받을 때 사용된다.
- 메시지를 받는 쪽에서 origin 검사를 제대로 하지 않았을 때, 취약점이 발생한다.
    - 특정 도메인만 허용한다고 표현식을 작성하고, 끝을 ‘$’로 고정하지 않거나 온점을 이스케이프 하지 않으면 공격자가 비슷한 도메인을 만들어 검사를 통과할 수 있다.

## Open Redirect 취약점

- 어떤 사이트가 url 파라미터를 받아서 다른 도메인으로 리다이렉트를 한다고 할 때, 정규 표현식을 잘못 쓰면 공격자 도메인으로 리다이렉트 될 수 있다.
    - 끝을 ‘$’로 고정하지 않거나 온점을 이스케이프 하지 않거나 하위 도메인을 너무 넓게 허용하면 발생한다.

## CORS 취약점

- 브라우저가 다른 도메인으로 요청을 보낼 때, 그 요청이 허용되는 지 확인하는 보안 메커니즘이다.
- 서버가 Origin 헤더를 정규 표현식으로 검사하고, 통과하면 Access-Control-Allow-Origin과 Access-Control-Allow-Credentials를 내려주는 경우가 있다. 이때 정규 표현식을 잘못 사용하면 공격자 도메인에서도 인증된 API 응답을 읽을 수 있다.
    - 위 취약점드들과 마찬가지로 점, 서브도메인 와일드카드, 끝 앵커 누락으로 인해 발생한다.

## SSRF 취약점

- 예시로 사용자가 LinkedIn 프로필 URL을 입력하면 서버가 해당 페이지를 가져와 PDF로 만들어주는 기능을 한다.
- 이것도 마찬가지로 잘못된 정규 표현식을 사용할 시 발생한다. 공격자가 자신이 가진 도메인을 LinkedIn처럼 보이게 만들거나, DNS를 내부 주소로 향하게 하거나, 리다이렉트를 이용해 서버가 내부망이나 클라우드 메타데이터 주소에 접근하게 만들 수도 있다.
    - 예방을 위해서는 허용할 스킴, 도메인, 경로까지 명확히 제한해야 한다.

## GitHub Recon

- 영상 후반부에서는 정규 표현식을 GitHub Recon에 활용하는 방법을 설명한다.
- 일반적인 서브도메인 탐색 도구만 사용하는 것이 아니라, GitHub에 올라온 코드나 설정 파일에서 정규 표현식으로 자산을 찾는 방식이다.
- 개발자가 실수로 GitHub에 올린 자동화 스크립트, 설정 파일, API 경로, 내부 도메인 등을 통해 새로운 공격 표면을 찾을 수 있다.

### 서브도메인 탐색

- 정규 표현식을 사용하면 GitHub에서 특정 회사 도메인과 관련된 서브도메인을 찾을 수 있다.
- 예를 들어 와일드카드 인증서 때문에 실제 서브도메인이 바로 보이지 않는 경우, GitHub에 노출된 코드나 설정에서 서브도메인을 찾을 수 있다.
- 단순히 전체 도메인을 찾는 것뿐만 아니라 `api`, `dev` 같은 키워드를 포함한 서브도메인을 찾으면 더 의미 있는 대상을 고를 수 있다.

### Context 확보

- GitHub Recon의 장점은 단순히 도메인 이름만 얻는 것이 아니라, 그 도메인이 어떻게 사용되는지도 알 수 있다는 점이다.
- 예를 들어 API 주소, 내부 경로, 자동화 방식, 요청 형식 같은 정보가 같이 노출될 수 있다.
- 영상에서는 이런 방식으로 일반적인 자산 탐색 도구에는 나오지 않던 내부 소스 코드 저장소 관련 정보를 발견한 사례를 설명한다.

## Content Discovery

- 정규 표현식은 API 경로나 숨겨진 엔드포인트를 찾는 데도 사용할 수 있다.
- 어떤 API에 Swagger나 OpenAPI 문서가 없더라도, GitHub에 올라온 코드에서 실제 사용되는 API 경로를 찾을 수 있다.
- 영상에서는 특정 서비스에 직접 계정을 만들 수 없는 상황에서도 GitHub에 노출된 코드에서 API 경로와 사용자명 생성 규칙을 찾은 사례를 설명한다.
- 즉, 정규 표현식은 접근하기 어려운 서비스의 내부 구조를 추측하거나 이해하는 데 도움을 줄 수 있다.

### API 경로 탐색

- GitHub에서 특정 회사 이름이나 도메인과 함께 `/api`, `/v1`, `/graphql` 같은 패턴을 찾을 수 있다.
- 이렇게 찾은 경로는 일반적인 브루트포싱으로는 발견하기 어려운 경우가 있다.
- 영상에서는 여러 단어가 조합된 복잡한 경로가 GitHub에 노출되어 있었고, 이를 통해 취약한 엔드포인트를 찾은 사례를 설명한다.

## GraphQL 탐색

- 정규 표현식은 GraphQL 엔드포인트를 찾는 데도 활용할 수 있다.
- GitHub에서 특정 도메인과 함께 GraphQL 관련 문자열을 찾으면, 해당 회사가 사용하는 GraphQL 인스턴스를 발견할 수 있다.
- 영상에서는 API뿐만 아니라 GraphQL, 특정 기술 스택, 내부 컴포넌트 등도 정규 표현식으로 찾을 수 있다고 설명한다.

## 데이터 덤프 파싱

- 영상에서는 Spring Boot Actuator와 heap dump 예시를 설명한다.
- heap dump는 메모리 스냅샷이기 때문에 사람이 바로 읽기 어렵다.
- 하지만 문자열을 추출한 뒤 정규 표현식으로 검색하면 의미 있는 값을 찾을 수 있다.

### 단순 grep의 한계

- 단순히 `password` 같은 단어만 찾으면 결과가 너무 많이 나온다.
- 관련 없는 변수명이나 로그가 많이 섞여서 중요한 값을 놓칠 수 있다.
- 정규 표현식을 사용하면 토큰, 쿠키, Base64 형태의 문자열, 파라미터처럼 보이는 값 등을 더 정확하게 찾을 수 있다.

## Secret 탐지

- 정규 표현식은 GitHub나 데이터 덤프에서 민감 정보를 찾는 데 사용할 수 있다.
- 영상에서는 JWT, AWS 키, Bearer 토큰 같은 값들이 일정한 패턴을 가지

### JWT

- JWT는 보통 특정한 형태로 시작한다.
- 이 특징을 이용하면 GitHub나 덤프 파일에서 JWT처럼 보이는 값을 찾을 수 있다.
- 단순 문자열 검색보다 패턴 기반 검색이 더 효과적이다.

### AWS Key, Bearer Token

- AWS 키나 Bearer 토큰도 특정한 문자열 구조를 가진다.
- 정규 표현식을 이용하면 이런 구조를 가진 값을 탐색할 수 있다.
- 영상에서는 이런 방식으로 과거에 놓쳤던 민감 정보를 더 많이 찾을 수 있었다고 설명한다.

## 코드 리뷰에서의 활용

- 정규 표현식은 코드 리뷰에도 사용할 수 있다.
- 특정 위험 함수나 사용자 입력이 직접 사용되는 부분을 빠르게 찾을 수 있다.

### XSS 가능성 탐지

- 영상에서는 PHP 예시를 사용한다.
- `GET` 또는 `POST`로 받은 값을 검증 없이 바로 출력하는 코드는 reflected XSS로 이어질 수 있다.
- 정규 표현식을 사용하면 이런 패턴을 코드 전체에서 빠르게 찾을 수 있다.

### RCE 가능성 탐지

- `system()` 같은 명령 실행 함수에 사용자 입력이 들어가는 경우도 위험하다.
- 정규 표현식으로 사용자 입력이 명령 실행 함수와 함께 쓰이는 부분을 찾으면 RCE 가능성을 검토할 수 있다.

### 코드 안의 정규 표현식 찾기

- 정규 표현식을 사용해서 코드 안에 있는 다른 정규 표현식을 찾을 수도 있다.
- PHP, JavaScript, Python 등에서 Regex를 사용하는 부분을 찾아 검토하면, 앞에서 설명한 잘못된 도메인 검증이나 앵커 누락 문제를 발견할 수 있다.

## Wordlist 생성

- 발표자는 정규 표현식을 사용해 GitHub에서 실제 개발자들이 사용하는 라우트와 파라미터를 수집할 수 있다고 설명한다.
- 이렇게 수집한 값은 정찰용 워드리스트로 사용할 수 있다.

### 라우트 수집

- Flask, Laravel, JavaScript, [ASP.NET](http://asp.net/), Ruby on Rails, Go 같은 프레임워크에서 라우트가 정의되는 방식은 어느 정도 패턴이 있다.
- 정규 표현식을 사용하면 GitHub에서 이런 라우트 정의를 추출할 수 있다.
- 추출한 라우트는 다른 대상의 경로 탐색에 활용할 수 있다.

### 파라미터 수집

- GET, POST 파라미터나 HTTP 헤더 이름도 GitHub에서 추출할 수 있다.
- 실제 개발자들이 사용하는 파라미터 이름을 모으면 일반적인 단어 리스트보다 현실적인 워드리스트를 만들 수 있다.

## 느낀점

정규 표현식을 리눅스에서 한 두번 사용해본 게 전부였는데 좀더 제대로된 개념을 알 수 있었다. 웹 취약점 쪽에서는 정말 중요하게 봐야되는 부분인 거 같다는 게 느껴졌다. 영어로 된 컴퍼런스 영상을 보는 건 처음이었는데 자동 번역 확장 프로그램이랑 gpt 같이 쓰면서 보니까 생각보다 볼 만했다. 이런 큰 컴퍼런스 내용은 당연히 엄청 어려울 거라고 생각했는데 내용도 웹 하나도 모르는 나도 어느정도 이해할 수 있는 난이도도 있다는 걸 알게되어서 흥미가 조금 생겼다!
